# Architecture: Feature #1 — Исправить расчёт штрафов при частичном погашении

## Проблема

Текущая система имеет три критических недостатка:

1. **Credit.debt не обновляется** — при создании `Payment` поле `Credit.debt` (остаток долга) не пересчитывается автоматически. Менеджер должен вручную обновлять кредит после каждого платежа.
2. **Сумма штрафа не валидируется** — при создании `Fine` менеджер вводит произвольную сумму, нет проверки что штраф не превышает актуальный остаток долга.
3. **date_constribution не сдвигается** — дата следующего платежа не пересчитывается после частичных оплат (закомментированный метод `getDateConstribution()` в Credit.php содержал эту логику, но был отключён).

---

## ADR (Architecture Decision Records)

### ADR-1: Где обновлять Credit.debt

**Context:**
Поле `Credit.debt` хранит остаток долга в БД. Сейчас оно не обновляется при создании/изменении/удалении платежа. Есть два подхода: (A) обновлять в контроллере `PaymentController`, (B) обновлять в модели `Payment::afterSave()` / `afterDelete()`.

**Decision:**
Использовать `Payment::afterSave()` и `Payment::afterDelete()` для автоматического пересчёта `Credit.debt`. Пересчёт — через метод `Credit::recalculateDebt()`, который вычисляет `debt = sum - SUM(payments.sum)`.

**Rationale:**
- Модельный подход гарантирует консистентность: любой код, создающий Payment (контроллер, консольная команда, тест), автоматически обновит debt.
- Контроллерный подход хрупкий — если Payment создаётся из другого места, debt рассинхронизируется.
- `recalculateDebt()` на основе `SUM(payments.sum)` устойчивее инкрементального `debt -= payment.sum`, т.к. самовосстанавливается при ошибках.

**Consequences:**
- (+) Автоматическая синхронизация debt при любом CRUD над Payment.
- (+) Один источник правды — `recalculateDebt()` в Credit.
- (-) Дополнительный SELECT при каждом сохранении Payment (минимальный overhead).
- (-) Нужно обработать edge case: удаление Payment тоже должно пересчитать debt.

---

### ADR-2: Валидация суммы штрафа

**Context:**
При создании `Fine` менеджер вводит сумму вручную. Нет проверки, что сумма штрафа адекватна: штраф может быть больше остатка долга или отрицательным.

**Decision:**
Добавить кастомный валидатор `Fine::validateFineSum()` в `rules()`. Валидатор проверяет:
1. `sum > 0` (уже покрыто правилом `['sum', 'number']`, но лучше явно `min`)
2. `sum` не превышает `Credit.debt` (опционально, как warning — бизнес может требовать штрафы больше долга)

**Rationale:**
- Inline-валидатор в модели — стандартный паттерн Yii2.
- Позволяет показать ошибку в форме до сохранения.
- Гибкость: если бизнес разрешает штрафы > долга, валидатор легко настроить.

**Consequences:**
- (+) Защита от ошибок ввода.
- (+) Пользователь видит ошибку в форме, а не после сохранения.
- (-) Нужен дополнительный запрос к Credit при валидации (1 SELECT).

---

### ADR-3: Пересчёт date_constribution

**Context:**
Закомментированный метод `Credit::getDateConstribution()` содержал логику сдвига даты следующего платежа на основе суммы оплат. Текущий `getDateConstributionStatus()` просто сравнивает фиксированную дату с today.

**Decision:**
Отложить на Phase 2. В текущей итерации (Feature #1) фокус на debt и fine validation. Пересчёт date_constribution — отдельная задача, т.к. требует уточнения бизнес-логики (что если платёж покрывает 1.5 месяца? как считать при досрочном погашении?).

**Rationale:**
- Минимизация scope: debt + fine validation — самодостаточные улучшения.
- Закомментированный код показывает, что логика пересчёта даты сложна и была отключена намеренно.

**Consequences:**
- (+) Меньший scope = меньше риск.
- (-) date_constribution остаётся ручным (как сейчас).

## ERD (изменения схемы БД)

**Изменений схемы не требуется.**

Все необходимые поля уже существуют:
- `credit.debt` (DOUBLE) — остаток долга
- `credit.sum` (DOUBLE) — тело долга
- `payment.sum` (DOUBLE) — сумма платежа
- `fine.sum` (FLOAT) — сумма штрафа
- `fine.id_credit` (INT) — FK на credit

**Рекомендация на будущее:** изменить типы `DOUBLE`/`FLOAT` на `DECIMAL(15,2)` для точности финансовых расчётов. Это отдельная миграция, не входит в scope Feature #1.

## API Contracts (затрагиваемые маршруты)

Все маршруты — стандартный Yii2 CRUD (не меняется интерфейс, только внутренняя логика):

| Маршрут | Метод | Изменение |
|---------|-------|-----------|
| `payment/create` | POST | После save — автоматический пересчёт Credit.debt (через afterSave) |
| `payment/update` | POST | После save — автоматический пересчёт Credit.debt (через afterSave) |
| `payment/delete` | POST | После delete — автоматический пересчёт Credit.debt (через afterDelete) |
| `fine/create` | POST | Валидация суммы штрафа перед сохранением |
| `fine/update` | POST | Валидация суммы штрафа перед сохранением |

**Контроллеры НЕ изменяются** — вся логика в моделях (afterSave, rules).

## Service Layer

**Отдельный Service-класс НЕ нужен.**

Обоснование:
- Логика простая: `debt = sum - SUM(payments.sum)`.
- Yii2 ActiveRecord callbacks (`afterSave`, `afterDelete`) — стандартный паттерн для side effects.
- В проекте нет service layer — вводить его ради одной фичи создаст inconsistency.
- Если в будущем логика усложнится (проценты, пени, графики), можно вынести в `CreditService`.

Метод `Credit::recalculateDebt()` инкапсулирует бизнес-логику пересчёта и может быть вызван из любого места.

## Последовательность операций (Flow)

### Создание платежа (Payment::afterSave)
```
1. Менеджер заполняет форму платежа (id_credit, sum, datetime)
2. PaymentController::actionCreate() вызывает $model->save()
3. Yii2 вызывает Payment::afterSave()
4. afterSave() загружает связанный Credit
5. Вызывает Credit::recalculateDebt()
6. recalculateDebt() считает SUM(payments.sum) и обновляет credit.debt
7. credit->save(false) — без валидации, т.к. меняем только debt
```

### Создание штрафа (Fine validation)
```
1. Менеджер заполняет форму штрафа (id_credit, sum, date)
2. Yii2 вызывает Fine::validate() -> rules() -> validateFineSum()
3. validateFineSum() загружает Credit, проверяет sum <= credit.debt
4. Если невалидно — добавляет ошибку, форма показывает сообщение
5. Если валидно — Fine::save() проходит
```

## Security Considerations

1. **Decimal precision:** Использовать `round($value, 2)` при всех финансовых расчётах. В будущем рекомендуется перейти на `bcmath` (`bcadd`, `bcsub` с scale=2) для точности.

2. **No raw SQL:** Все запросы через ActiveRecord / Query Builder. Метод `recalculateDebt()` использует `Payment::find()->where()->sum('sum')`.

3. **Race conditions:** При одновременном создании двух платежей возможен race condition на обновлении debt. Для текущей нагрузки (single-user OSPanel) это не критично. Для production рекомендуется:
   - `SELECT ... FOR UPDATE` при пересчёте
   - Или оптимистичная блокировка через `optimistic_lock` в Credit

4. **Отрицательный debt:** Добавить проверку `debt >= 0` после пересчёта. Если сумма платежей > суммы кредита — это ошибка данных.

5. **CSRF:** Уже защищено через Yii2 `VerbFilter` (POST для create/update/delete).

6. **Авторизация:** Текущий код не проверяет права доступа в контроллерах. Это существующая проблема, не входит в scope Feature #1.

## Риски и митигации

| Риск | Вероятность | Митигация |
|------|-------------|-----------|
| Существующие данные: debt рассинхронизирован | Высокая | Одноразовый скрипт пересчёта для всех кредитов |
| afterSave() падает — Payment сохранён, debt нет | Низкая | Обернуть в транзакцию |
| Удаление Payment без afterDelete | Средняя | Реализовать afterDelete() |
