# Feature #1: Исправить расчёт штрафов при частичном погашении

**Issue:** #1  
**Status:** Analysis  
**Date:** 2026-04-15  
**Author:** analyst-agent

---

## Описание

При частичном погашении кредита поле `Credit.debt` (остаток долга) не пересчитывается автоматически при сохранении платежа. Штрафы создаются вручную через CRUD без привязки к актуальному остатку долга, что приводит к завышенной или заниженной базе начисления штрафа.

**Корневая причина:**
1. `Payment::save()` не обновляет `Credit.debt`.
2. При создании `Fine` менеджер вводит сумму вручную — нет привязки к актуальному `Credit.debt`.

**Ожидаемое поведение:** база штрафа всегда равна `Credit.debt` на момент нарушения срока. `Credit.debt` уменьшается на `Payment.sum` при каждом сохранении платежа.

---

## Functional Requirements

| ID | Требование |
|----|-----------|
| FR-1 | При успешном сохранении нового `Payment` система автоматически уменьшает `Credit.debt` на величину `Payment.sum`. |
| FR-2 | `Credit.debt` не может стать отрицательным: если `Payment.sum > Credit.debt`, в `Credit.debt` записывается `0`. |
| FR-3 | При создании записи `Fine` форма отображает текущее значение `Credit.debt` как предзаполненную базу для расчёта штрафа. |
| FR-4 | Сохранение `Fine` допускается только при `Fine.sum > 0` и при наличии связанного кредита с `Credit.debt > 0`. |
| FR-5 | История платежей (`Payment` list) для кредита отображает актуальное `Credit.debt` после каждого платежа. |

---

## Non-Functional Requirements

| ID | Требование |
|----|-----------|
| NFR-1 | Обновление `Credit.debt` должно выполняться в одной транзакции с сохранением `Payment` — не допускать рассинхронизации при ошибке. |
| NFR-2 | Изменение не должно ломать существующие отчёты, использующие `Credit.debt` и `Credit.getPayment()`. |
| NFR-3 | Покрытие unit-тестами (Codeception) граничных случаев: полное погашение, переплата, нулевой остаток. |
| NFR-4 | Обратная совместимость: существующие записи `Fine` и `Payment` не пересчитываются автоматически при деплое. |

## Acceptance Criteria

### AC-1: Остаток долга уменьшается пропорционально платежу

**Given** кредит с `debt = 1000`, по которому ещё нет платежей  
**When** кассир создаёт платёж на сумму `400`  
**Then** `Credit.debt` становится `600`, платёж успешно сохранён

---

### AC-2: Штраф начисляется только на фактический остаток

**Given** кредит с `debt = 600` (после частичного погашения)  
**When** менеджер открывает форму создания штрафа для этого кредита  
**Then** поле базы штрафа предзаполнено значением `600` (не исходным `sum`)

---

### AC-3: История платежей корректно отражает уменьшение штрафной базы

**Given** кредит с `debt = 1000` и двумя последовательными платежами — `300` и `200`  
**When** менеджер открывает список платежей по кредиту  
**Then** текущий `Credit.debt` равен `500`, в истории видны оба платежа

---

### AC-4: Граничный случай — платёж превышает остаток

**Given** кредит с `debt = 100`  
**When** кассир вводит платёж на сумму `150`  
**Then** `Credit.debt` становится `0` (не отрицательным), платёж сохранён

## Out of Scope

- Пересчёт `date_constribution` при каждом платеже.
- Интеграция с внешними платёжными системами.
- Изменение UI страниц просмотра кредита (только форма создания штрафа).

## Technical Notes

### Затронутые файлы

| Файл | Изменение |
|------|-----------|
| `models/Payment.php` | Добавить `afterSave()` — уменьшать `Credit.debt` на `$this->sum` внутри транзакции. |
| `models/Credit.php` | Метод `getPayment()` остаётся корректным (`sum - debt`), изменений не требует. |
| `models/Fine.php` | Добавить виртуальный атрибут или scenario для предзаполнения из `Credit.debt`. |
| `controllers/FineController.php` | В `actionCreate()` передавать `$credit->debt` в view при наличии `id_credit` в GET-параметрах. |
| `controllers/PaymentController.php` | Проверить, что `actionCreate()` использует транзакцию (или делегировать в модель). |
| `tests/unit/models/` | Новый файл `CreditDebtTest.php` с тестами граничных случаев. |

### Ключевые технические решения

- Использовать Yii2 `afterSave($insert, $changedAttributes)` в `Payment` для обновления `Credit.debt`.
- Обновление через `Credit::updateAllCounters(['debt' => -$this->sum], ['id' => $this->id_credit])` — атомарная операция.
- Если `debt - sum < 0`, использовать `UPDATE credit SET debt = 0 WHERE id = :id AND debt < :sum`.
- Для `Fine` форма: передать `$credit` в view, предзаполнить поле `sum` через `$model->sum = $credit->debt`.

### Схема БД (без изменений)

```
credit: id, sum, debt, date_constribution, ...
payment: id, id_credit, sum, datetime, note
fine: id, id_credit, sum, date, note
```

### Зависимости между stories

```
Story A (Payment → debt) → Story B (Fine → debt) → Story C (Tests)
```
Story B зависит от Story A: тесты штрафов должны проверять корректный `debt`.
