# Feature #23: Восстановить выбор даты следующей оплаты при ежемесячном платеже

## Описание

При проведении ежемесячного платежа по кредиту пользователь должен иметь возможность вручную указать дату следующей оплаты (`date_constribution`). В текущей реализации дата всегда вычисляется автоматически как `date_constribution + 1 месяц`, и пользователь не может её изменить. Необходимо добавить поле ввода даты в форму ежемесячного платежа, предзаполнять его автоматически вычисленной датой, но разрешать пользователю изменять значение перед сохранением.

## Functional Requirements

**FR-1. Отображение поля даты следующей оплаты**
В модальном окне (или форме) ежемесячного платежа должно присутствовать поле для ввода даты следующей оплаты (`date_constribution`). Поле должно быть обязательным к заполнению перед отправкой.

**FR-2. Предзаполнение поля автоматически вычисленной датой**
При открытии формы ежемесячного платежа поле даты должно быть предзаполнено датой, равной текущему значению `date_constribution` кредита плюс один календарный месяц (то же значение, которое вычисляет `Credit::recalculateNextPaymentDateFromMonth()`).

**FR-3. Передача выбранной даты на сервер**
При отправке формы выбранная пользователем дата должна передаваться в AJAX-запросе `$.get('payment-month', ...)` как отдельный параметр `date_constribution`.

**FR-4. Использование переданной даты на сервере**
`CreditController::actionPaymentMonth()` должен принимать параметр `date_constribution`. Если параметр передан и корректен, значение `date_constribution` модели Credit должно быть установлено в переданную дату, а не вычислено автоматически. Если параметр отсутствует или пуст, применяется автоматическое вычисление (обратная совместимость).

**FR-5. Сохранение выбранной даты**
После успешного проведения платежа поле `date_constribution` в записи кредита должно содержать именно ту дату, которую выбрал пользователь (или автоматически вычисленную, если пользователь не изменял значение).

**FR-6. Валидация даты на сервере**
Переданная дата должна быть валидной датой в формате `Y-m-d`. Дата не должна быть раньше текущей даты платежа. При невалидном значении операция должна завершаться с ошибкой без изменения данных.

## Non-Functional Requirements

**NFR-1. Совместимость с Bootstrap 3**
Элемент ввода даты должен использовать виджет или нативный `<input type="date">`, стилизованный под Bootstrap 3 (классы `form-control`, `form-group`).

**NFR-2. Минимальные изменения в архитектуре**
Изменения должны затрагивать только `views/credit/view_credit.php` (или соответствующий partial), `web/js/main.js` и `controllers/CreditController.php`. Модель `Credit` изменяется только при необходимости дополнительной валидации.

**NFR-3. Обратная совместимость**
Параметр `date_constribution` в `actionPaymentMonth()` должен быть необязательным (`$date_constribution = null`), чтобы не сломать существующие вызовы (например, консольные команды или другие интеграции).

**NFR-4. PHP 7.1 совместимость**
Весь новый серверный код должен быть совместим с PHP 7.1: без union types, nullable types записываются через `?тип` или значение по умолчанию `null`.

## Acceptance Criteria

### AC-1: Поле даты отображается в форме ежемесячного платежа

**Given** пользователь открывает карточку кредита и инициирует проведение ежемесячного платежа (нажимает соответствующую кнопку/ссылку)
**When** форма (модальное окно) ежемесячного платежа отображается на экране
**Then** в форме присутствует поле ввода даты с меткой «Дата следующей оплаты», предзаполненное датой равной `date_constribution` кредита + 1 месяц, и пользователь может изменить это значение.

### AC-2: Пользователь выбирает другую дату и платёж сохраняется с ней

**Given** форма ежемесячного платежа открыта и поле даты предзаполнено автоматически вычисленной датой
**When** пользователь изменяет дату на произвольную корректную дату в будущем и подтверждает платёж
**Then** AJAX-запрос содержит выбранную дату, платёж успешно проводится, и значение `date_constribution` в БД для данного кредита равно дате, выбранной пользователем.

### AC-3: Пользователь не изменяет дату — используется автоматически вычисленная

**Given** форма ежемесячного платежа открыта и поле даты предзаполнено автоматически вычисленной датой (`date_constribution + 1 месяц`)
**When** пользователь не изменяет значение поля и подтверждает платёж
**Then** платёж успешно проводится, и значение `date_constribution` в БД равно автоматически вычисленной дате (как при старом поведении).

### AC-4: Невалидная дата приводит к ошибке без изменения данных

**Given** форма ежемесячного платежа открыта
**When** пользователь вводит некорректную дату (например, прошедшую дату раньше текущей даты платежа или неверный формат) и подтверждает платёж
**Then** сервер возвращает ошибку валидации, данные кредита в БД не изменяются, платёж не проводится, пользователю отображается сообщение об ошибке.

### AC-5: Предзаполнение корректно для кредитов с разными датами

**Given** несколько кредитов с разными значениями `date_constribution` (в том числе в конце месяца, например 31-е число)
**When** пользователь открывает форму ежемесячного платежа для каждого кредита
**Then** поле даты предзаполнено корректной датой: `date_constribution + 1 месяц` с учётом граничных случаев (например, 31 января → 28/29 февраля), аналогично результату `Credit::recalculateNextPaymentDateFromMonth()`.

## Out of Scope

- Изменение логики автоматического вычисления даты в методе `Credit::recalculateNextPaymentDateFromMonth()`.
- Добавление выбора даты следующей оплаты при других типах платежей (частичное погашение, штраф и т.д.).
- Уведомления или напоминания по дате следующей оплаты.
- История изменений даты `date_constribution`.
- Ограничение выбора даты через UI (например, запрет выбора прошедших дат на уровне datepicker) — только серверная валидация.
- Изменение структуры БД (новых столбцов не добавляется).

## Technical Notes

- **Точка входа JS:** функция `receivedCredit(id)` в `web/js/main.js` формирует AJAX-запрос `$.get('payment-month', ...)`. Необходимо считать значение нового поля и добавить его как параметр `date_constribution`.
- **Контроллер:** `CreditController::actionPaymentMonth($sum, $note, $id_credit)` — добавить необязательный параметр `$date_constribution = null`. Если он передан и валиден, установить `$model->date_constribution = $date_constribution` до вызова `$model->save()`. Иначе вызвать существующий `$model->recalculateNextPaymentDateFromMonth()`.
- **View:** `views/credit/view_credit.php` — в форме/блоке ежемесячного платежа добавить `<input type="date" class="form-control" id="month_payment_date">`. Предзаполнение выполняется через JS при открытии формы: получить текущее `date_constribution` кредита (можно передать через `data`-атрибут или отдельный скрытый элемент) и добавить к нему 1 месяц.
- **Формат даты:** стандарт `Y-m-d` (MySQL DATE), нативный `<input type="date">` работает с этим форматом.
- **Валидация сервера:** использовать `\DateTime::createFromFormat('Y-m-d', $date_constribution)` для проверки корректности даты перед присвоением модели.
- **Связанные файлы:**
  - `controllers/CreditController.php` — `actionPaymentMonth()`
  - `models/Credit.php` — `recalculateNextPaymentDateFromMonth()`
  - `views/credit/view_credit.php`
  - `web/js/main.js` — `receivedCredit()`
